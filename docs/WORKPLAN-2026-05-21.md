# Rafiki Work Plan — 2026-05-21

This plan follows PR #124, which tightens portal quality, archive health, export
metadata stamping, visual E2E baselines, and card-level evaluation state.

## Current State

- The portal is still local-first and static-friendly.
- Review mode now carries ratings, feedback, metadata, evaluation decisions,
  lineage chips, Review Queue, and run-level decision summaries.
- Teach mode has a Curriculum Atlas, facilitator prompts, critique criteria,
  and a concept graph.
- `archive-health` is read-only and reports missing images, malformed manifests,
  duplicate filenames, orphaned sidecar state, disk usage, and cleanup risk.
- `npm run smoke:dry-run` now proves the spend-free Node CLI, MCP status/tool
  listing, MCP dry-run bridge, and archive-health path against disposable data.
- The renderer split has started with Atlas helpers, but
  `lib/renderers/library.py` remains the biggest concentration of UI code.

## Next Execution Lanes

### 1. Conservative Archive Cleanup Report

Build on `archive-health` without deleting anything:

- Group cleanup-risk findings by project and run.
- Label findings as safe-to-ignore, needs-human-review, or repair-first.
- Print exact follow-up commands for rebuild, approve, export, or manual review.
- Keep mutation behind a separate future command only after the report contract
  has settled.

### 2. Static Deploy Source Mapping

Make deployed artifacts trace back to source cards more reliably:

- Stamp metadata for project-level deploys that point at approved viewers.
- Detect static deploys that cannot map to run images and report why.
- Add tests for approved, run-viewer, and unmapped deploy cases.

### 3. Evaluation Meets Curriculum

Turn evaluation from a local note into a teaching instrument:

- Let cards reference Atlas modules or critique criteria.
- Surface "why this fits" and "what to discuss" beside evaluation decisions.
- Summarize decision/score patterns by module in Teach mode.

### 4. Visual Review Artifacts

Keep the current metric-based E2E baseline as the fast gate, then add optional
human artifacts:

- Keep screenshots with `RAFIKI_E2E_KEEP_TMP=1`.
- Add named desktop Review, desktop Teach, and mobile captures to the E2E JSON.
- Consider a future reviewed baseline directory only after layout churn slows.

## Documentation Hygiene Rules

- Active agent-facing truth lives in `README.md`, `docs/INDEX.md`,
  `docs/ROADMAP.md`, `docs/PORTAL-COMMAND-CENTER.md`, and focused docs under
  `docs/`.
- Historical audit files may describe past gaps, but each should now carry
  follow-up notes when a gap has shipped.
- Local state files under `output/` and `data/` stay gitignored unless an
  operator intentionally exports a clean artifact.
- Avoid new frameworks until the portal renderer split proves where a frontend
  boundary would actually pay for itself.
